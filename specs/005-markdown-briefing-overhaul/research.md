# Research: Markdown Briefing Overhaul

**Feature**: 005-markdown-briefing-overhaul  
**Date**: 2026-04-26

## Research Tasks

### R1: Markdown rendering approach — string builder vs template engine

**Decision**: Plain string concatenation (f-strings + list join)

**Rationale**: The markdown output is a flat sequence of headings,
bullet lists, and paragraphs. There is no conditional nesting, loops
over nested structures, or partial templates. Python f-strings with a
list of sections joined by `\n\n` is the simplest approach. A template
engine (Jinja2, Mako) would add a dependency and indirection for no
benefit — violating Constitution Principle I (Simplicity First).

**Alternatives considered**:
- Jinja2 templates: More structure, but requires learning template
  syntax, adds a dependency, and the output is simple enough that
  templates add no value.
- `io.StringIO` buffer: Marginally useful for large outputs, but
  markdown briefings are <50KB. List-join is simpler.

### R2: Task grouping by Area — data flow

**Decision**: Group tasks by Area in the `render_markdown` tool handler
using `itertools.groupby` on the already-sorted task list. No changes
needed to the Things integration or Task model — the `area` field
already exists on Task and is already populated by `things.py`.

**Rationale**: The `area` field was added to the Task model in the
001 feature and the Things integration already resolves it from the
SQLite database (including project→area inheritance). The only gap is
that the task tool handlers (`tools/tasks.py`) omit `area` from the
JSON output — adding it is a one-line change per handler.

**Alternatives considered**:
- Group in the agent: The agent could group tasks client-side, but
  that pushes presentation logic into the prompt and makes the
  markdown output dependent on the agent's formatting. Grouping in
  the renderer keeps it deterministic and testable.
- New MCP tool for grouped tasks: Unnecessary complexity. The existing
  tools just need to include the `area` field.

### R3: Action suggestions — agent-generated vs server-generated

**Decision**: Action suggestions are generated by the LLM agent and
passed to `render_markdown` as structured data (list of dicts with
`task_title` and `suggestion` fields). The MCP server does not generate
suggestions.

**Rationale**: The value of action suggestions comes from the LLM's
reasoning about the task title and context. The MCP server has no LLM
access. The agent already has the task list from Step 2, can filter
for null-area tasks, randomly select up to 5, generate suggestions,
and pass them as an additional parameter to `render_markdown`.

**Alternatives considered**:
- Server-side suggestion generation: Would require adding an LLM
  client to the MCP server, violating the architecture (the server
  is a data tool, not an AI agent).
- Separate MCP tool for suggestions: Adds API surface for something
  the agent can do inline. YAGNI.

### R4: Random selection of unassigned tasks

**Decision**: The agent performs random selection from the unassigned
task list. The `render_markdown` tool receives the already-selected
tasks with suggestions — it does not need to know about the selection
algorithm.

**Rationale**: The agent has the full task list and can use its own
randomness. The MCP server's `render_markdown` tool should be a pure
renderer that takes structured data and writes markdown. Selection
logic belongs in the orchestrator (agent), not the renderer.

**Alternatives considered**:
- Server-side random selection: Would couple selection logic to the
  renderer. If the selection criteria change (e.g., "oldest 5" instead
  of "random 5"), the server code must change. Keeping it in the agent
  makes it a prompt change only.

### R5: Removing ReportLab — impact assessment

**Decision**: Remove `reportlab>=4.1` from `pyproject.toml` dependencies.
Delete the entire `src/daily_planner/pdf/` package (4 files). Delete
`tests/integration/test_render_pdf.py`. Delete
`src/daily_planner/tools/render_pdf.py`.

**Rationale**: The PDF pipeline is being fully replaced by markdown.
There is no backward-compatibility requirement for PDF output (per
clarification: the tool is renamed, not aliased). Keeping dead code
violates Principle I.

**Impact**:
- `pyproject.toml`: Remove `reportlab>=4.1` from dependencies
- `src/daily_planner/pdf/`: Delete entire package (renderer.py,
  page_one.py, page_two.py, __init__.py)
- `src/daily_planner/tools/render_pdf.py`: Delete (replaced by
  render_markdown.py)
- `tests/integration/test_render_pdf.py`: Delete (replaced by
  test_render_markdown.py)
- `server.py`: Rename tool registration
- No other modules import from `pdf/` or `render_pdf`

### R6: Configuration simplification

**Decision**: Remove `[page_one]` and `[page_two]` TOML sections and
the `page_one_font_size` / `page_two_font_size` fields from the
Configuration dataclass. The `[output]` section remains with `path`
and `repos_file`.

**Rationale**: Font sizes are PDF-specific. Markdown has no font size
concept. The Configuration model becomes simpler (2 fields instead of 4).

**Impact on config loader**: `load_configuration()` drops the two
`page_one`/`page_two` dict lookups. Existing `settings.toml` files
with those sections still parse without error (TOML ignores unknown
sections when you don't read them), so no migration needed.

### R7: `render_markdown` tool contract — new parameters

**Decision**: The `render_markdown` tool adds one new optional parameter:
`action_suggestions: list[dict] | None`. Each dict has `task_title: str`
and `suggestion: str`. All other parameters remain the same as
`render_pdf` (minus the font-size-related Configuration fields which
are internal).

**Rationale**: The action suggestions section is conditional (omitted
when empty/null). Making it an optional parameter keeps the tool
backward-compatible with agents that don't pass it. The response
changes from `{"pdf_path": ..., "pages": 2}` to
`{"markdown_path": ...}`.

### R8: Agent instructions update scope

**Decision**: Update `.github/agents/morning-briefing.agent.md` to:
1. Change description from "PDF" to "markdown"
2. Add Step 5.5: "Generate Action Suggestions" between summarize and render
3. Update Step 6: Call `render_markdown` instead of `render_pdf`, include
   `action_suggestions` parameter
4. Add formatting guidance in Step 5: "Separate themes with blank lines"
5. Update Step 7: Report markdown path instead of PDF path

**Rationale**: The agent file is the orchestrator. It needs to know
about the new workflow step (action suggestions) and the renamed tool.

### R9: Date format in markdown filename

**Decision**: Use `morning-briefing-YYYY-MM-DD.md` format
(e.g., `morning-briefing-2026-04-26.md`).

**Rationale**: The spec (FR-004) specifies this format. It differs from
the old PDF filename format (`YYYY-MM-DD dddd.pdf`) but the spec
explicitly defines the new convention. The date-first format makes files
sort chronologically in directory listings.
